## 一、产品概述

### 1.1 产品背景

当前园区网络运维存在以下典型痛点：

- **多系统割裂**：NetBox（资产管理）、Zabbix（监控告警）、ELK（日志分析）等系统独立，运维人员需要频繁切换系统
- **告警多、日志杂**：海量的告警和日志，问题定位依赖个人经验
- **重复性操作**：大量重复性运维工作（巡检、备份、配置对比、健康度检查）
- **知识沉淀不足**：运维知识难以积累和传承，新人上手成本高
- **缺乏自动化工具**：缺乏系统化的自动化运维工具

### 1.2 产品定位

> **面向园区网络运维的 AI 驱动统一自动化平台**
>
> 通过 **对话式交互 + 自动化任务调度**，由 AI 自动识别用户意图、调度工具、执行巡检、备份、健康度检查等运维任务，实现运维工作的智能化和自动化。

### 1.3 目标用户

| 角色 | 使用目标 |
|------|----------|
| 网络运维工程师 | 快速定位问题、执行自动化任务、减少手工操作 |
| 运维负责人 | 掌握整体运行状态、风险趋势、自动化执行情况 |
| 新手运维 | 通过对话完成复杂运维任务，降低学习成本 |

---

## 二、总体设计原则

1. **AI 负责调度与分析，工具负责执行**
2. **所有执行动作可审计、可回放、可追溯**
3. **先覆盖 80% 高频场景，再做高级自动化**
4. **前端以"运维工作台 + 各功能模块"为核心**
5. **安全性优先：只读操作为主，可执行操作需确认**

## 四、功能模块设计

### 4.1 运维工作台（Dashboard）

**功能说明**：

提供 AI 对话式运维交互界面，用户通过自然语言输入运维需求，AI 自动识别意图、生成执行计划、调度工具、汇总结果。

**核心功能**：

- **会话管理**：支持多会话切换，保存历史对话
- **意图识别**：自动识别用户查询意图（查询、巡检、分析、诊断等）
- **执行计划生成**：根据意图自动生成执行步骤
- **执行轨迹可视化**：实时展示执行步骤和结果
- **结果总结**：汇总执行结果，生成自然语言回复

**已实现功能**：

- ✅ 三栏布局：会话列表、对话区、执行轨迹
- ✅ AI 对话交互
- ✅ 执行步骤可视化
- ✅ 会话历史管理
- ✅ 意图识别（IntentAgent）
- ✅ 执行计划生成（PlannerAgent）
- ✅ 执行控制（ExecutorAgent）
- ✅ 结果总结（Summarizer）

**示例场景**：

**场景 1：设备健康度查询**
> 用户输入："昨天核心交换机有没有异常？"

AI 行为拆解：
1. 识别对象：核心交换机（NetBox）
2. 时间范围：昨天
3. 查询 Zabbix：CPU / 内存 / 端口状态
4. 查询 ELK：设备日志
5. 汇总分析 → 输出健康度报告

**场景 2：告警分析**
> 用户输入："分析一下最近的告警趋势"

AI 行为拆解：
1. 识别意图：告警趋势分析
2. 查询 Zabbix：最近告警数据
3. 聚合分析：按时间、设备、类型统计
4. 生成趋势图表
5. 输出分析结论

### 4.2 资产视图（Assets）

**功能说明**：

集成 NetBox 资产管理系统，提供设备、IP、站点、机柜、VLAN 等资产管理功能。

**核心功能**：

- **设备管理**：设备列表查询、设备详情查看、设备配置查看
- **IP 管理**：IP 地址查询、IP 使用率统计
- **站点管理**：站点列表查询、站点详情查看
- **机柜管理**：机柜列表查询、机柜详情查看（含设备布局）
- **VLAN 管理**：VLAN 列表查询、VLAN 详情查看
- **设备标签**：显示设备标签（stack-master、stack-member 等）

**已实现功能**：

- ✅ 设备列表查询（支持按名称、站点、角色过滤）
- ✅ 设备详情查看（类似机柜详情的布局）
- ✅ 设备标签显示（stack-master 金色标签，stack-member 灰色标签）
- ✅ 设备配置查看（支持从设备获取配置）
- ✅ 设备配置获取（SSH 连接，支持华为设备分页处理）
- ✅ IP 地址查询
- ✅ 站点查询
- ✅ 机柜查询（138 个机柜）
- ✅ VLAN 查询（214 个 VLAN）
- ✅ 带有 IP 的设备查询（273 台设备）

**API 端点**：

| 方法 | 端点 | 说明 |
|------|------|------|
| GET | `/api/assets/devices` | 查询设备列表 |
| GET | `/api/assets/devices/with-ip` | 查询有 IP 的设备 |
| GET | `/api/assets/ips` | 查询 IP 地址 |
| GET | `/api/assets/sites` | 查询站点 |
| GET | `/api/assets/racks` | 查询机柜 |
| GET | `/api/assets/vlans` | 查询 VLAN |

**技术实现**：

- 使用 NetBox API 获取资产数据
- 使用 Paramiko 进行 SSH 连接获取设备配置
- 配置存储在 NetBox 的 `local_context_data` 字段
- 支持华为设备的 "More" 分页处理

### 4.6 自动化任务模块（待重构）

**状态**：已移除，待 Phase 2 重新设计

**原因**：

原自动化任务模块（配置备份、网络巡检、健康度检查）存在以下问题：

1. 架构设计不合理，与 MCP 层耦合过紧
2. 任务执行逻辑复杂，难以维护
3. 缺乏统一的任务调度机制
4. 缺乏任务状态管理和监控

**计划**：

在 Phase 2 中重新设计自动化任务模块，采用以下原则：

1. 任务与 MCP 层解耦
2. 统一的任务调度框架
3. 完善的任务状态管理
4. 任务执行监控和告警

**Phase 2 规划**：

详见"五、实施计划"章节。

## 六、非功能性需求

### 6.1 性能要求

- 查询响应时间 < 5s
- AI 分析响应时间 < 15s
- 巡检任务执行时间 < 30min（100 台设备）
- 支持并发用户数 > 50

### 6.2 可扩展性

- 工具统一 MCP 接口，新系统接入无需改前端
- 支持自定义巡检模板
- 支持自定义告警规则
- 支持插件化扩展

### 6.3 可运维性

- 所有 AI 行为可回溯
- 每次调度生成唯一 Trace ID
- 任务执行日志完整记录
- 支持日志审计和查询

### 6.4 安全性

- 敏感信息加密存储
- 操作日志审计
- 权限控制（RBAC）
- 配置备份版本管理
- API 访问频率限制

### 6.5 可用性

- 系统可用性 > 99%
- 故障恢复时间 < 30min
- 数据备份频率：每日
- 支持热备份

## 八、风险和依赖

### 8.1 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 设备连接不稳定 | 自动化任务失败 | 实现重试机制、超时处理 |
| 配置文件过大 | 备份/对比慢 | 分块处理、异步执行 |
| AI 分析不准确 | 误报/漏报 | 持续优化 Prompt、人工审核 |
| 外部系统 API 变化 | 集成失败 | 版本锁定、API 兼容性测试 |

### 8.2 依赖项

- NetBox API 稳定
- Zabbix API 稳定
- ELK API 稳定
- 本地大模型可用

### 8.3 业务风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 用户接受度低 | 使用率低 | 用户培训、持续优化 |
| 运维流程变更 | 工作量增加 | 分阶段上线、逐步迁移 |
| 数据安全 | 数据泄露 | 加密存储、权限控制 |
