# AgenticOps v0.2.0 Release Design

## Objective

Publish the first durable multi-agent diagnostic release as `v0.2.0`. The GitHub Release, annotated Git tag and remote default branch must point to the same verified commit on `main`.

## Selected release path

The release will be prepared on `codex/multi-agent-core`, pushed for traceability, then fast-forwarded into local `main`. After the complete release gate passes on `main`, push `main`, create an annotated `v0.2.0` tag and publish a non-draft, non-prerelease GitHub Release.

Alternatives were rejected for these reasons:

- Tagging the feature branch would leave the default branch behind the published source.
- Opening a PR and waiting for manual merge adds an external pause the user did not request; the owner explicitly authorized direct publication.

No force push, history rewrite or deletion of existing tags is allowed.

## Documentation and version scope

The release changes documentation and metadata only where required for an accurate v0.2.0 handoff:

- Update backend and frontend version metadata to `0.2.0`.
- Update both README files with asynchronous Graph semantics, quick-start, upgrade, verification, safety boundaries and links to detailed operations documents.
- Make `DEPLOYMENT.md` a complete install/upgrade path for the new API/Worker split, including backup, migration, health, Observe-only and rollback steps.
- Align `docs/PRODUCTION_DEPLOYMENT.md`, `docs/RUNBOOK.md` and release precheck instructions with Graph Worker, Checkpoint and new endpoints.
- Add bilingual-quality `docs/RELEASE_NOTES_v0.2.0.md` content suitable for the GitHub Release body.

Documentation must not claim real device, NetBox, ELK, Zabbix or production LLM end-to-end validation. It must state that autonomous high-risk execution remains disabled.

## Release gate

The exact release commit must pass:

1. Clean Git worktree and no placeholder scan failures.
2. Backend full test suite against PostgreSQL.
3. Alembic head check and downgrade/upgrade round trip.
4. Ruff blocking checks, Python compile, Bandit high-severity/high-confidence gate and pip-audit.
5. Frontend type-check and production build.
6. Docker Compose config, image build, empty-database migration, backend/worker health and frontend HTTP check.
7. Container verification that `AUTOMATION_OBSERVE_ONLY=True`, OpenAI/httpx versions are compatible and an empty API key does not eagerly initialize the external client.

External integrations remain empty or use controlled fake adapters during release validation.

## Publication sequence

1. Commit documentation and version updates on `codex/multi-agent-core`.
2. Push `codex/multi-agent-core` to `origin`.
3. Fast-forward local `main` to the verified release commit; abort if it is not a fast-forward.
4. Push `main` to `origin` without force.
5. Confirm GitHub Actions status for the pushed `main` commit. If checks fail, stop publication, fix on the feature branch, and repeat the gate.
6. Create annotated tag `v0.2.0`, push it, and publish the GitHub Release from `docs/RELEASE_NOTES_v0.2.0.md`.
7. Verify the remote tag, Release URL, target commit and published/non-prerelease state.

## Failure and rollback behavior

- Documentation/test failure: do not push or tag.
- Feature branch push failure: leave local commits intact and report the remote error.
- Main fast-forward or push conflict: stop; never force push.
- CI failure on main: do not create the release tag.
- Tag push succeeds but Release creation fails: keep the immutable tag, retry Release creation, and report the partial state.
- Release metadata error: edit the Release body/title; do not move the tag after publication.

Database rollback remains backup restoration for production. Alembic downgrade is validated for development but is not the primary production recovery procedure.

## Success criteria

- `origin/main`, `refs/tags/v0.2.0`, and GitHub Release `v0.2.0` resolve to the same commit.
- Release is published, not draft, and not prerelease.
- README badges/API metadata/frontend metadata report v0.2.0.
- Deployment and upgrade instructions are complete and safety defaults remain Observe-only.
- The repository and Release notes clearly distinguish implemented, fake-tested and not-yet-implemented capabilities.
