# 异常跟踪器改进说明

## 问题描述

之前的自动化处理逻辑存在以下问题：

1. **触发过于频繁**：每5分钟采样一次，每次采样都会触发完整的研判流程
2. **没有累积机制**：单次异常就触发处理，没有考虑异常的持续性
3. **没有去重机制**：同一类型的异常会重复创建任务
4. **任务数量过多**：用户容易忽略或感到疲劳

## 解决方案

### 1. 异常累积机制

同一台设备的同一种异常需要累积到一定次数才触发研判：

- **配置参数**：`accumulation_threshold`（默认：3次）
- **工作原理**：
  - 每次采样检测到异常时，增加该设备该异常类型的计数
  - 只有当计数达到阈值时才触发研判
  - 触发后重置计数

**示例**：
- 设备A的接口flap异常需要连续3次采样都超过阈值才触发研判
- 如果只有1次或2次，只增加计数，不触发研判

### 2. 时间窗口去重

在一定时间窗口内，同一台设备的同一种异常只触发一次研判：

- **配置参数**：`dedup_window_minutes`（默认：30分钟）
- **工作原理**：
  - 记录每次触发研判的时间
  - 在时间窗口内，同一设备同一异常类型只触发一次
  - 超过时间窗口后可以再次触发

**示例**：
- 设备A的接口flap异常在10:00触发了一次研判
- 在10:00-10:30期间，即使再次检测到异常，也不会再次触发
- 10:30之后可以再次触发

### 3. 冷却时间

同一台设备的同一种异常触发研判后，需要等待一段时间才能再次触发：

- **配置参数**：`cooldown_minutes`（默认：60分钟）
- **工作原理**：
  - 触发研判后，进入冷却期
  - 冷却期内不会再次触发研判
  - 冷却期结束后，重新开始累积计数

**示例**：
- 设备A的接口flap异常在10:00触发了一次研判
- 在10:00-11:00期间，即使检测到异常，也不会触发研判
- 11:00之后，重新开始累积计数，需要连续3次异常才再次触发

### 4. 异常状态跟踪

在内存中跟踪每台设备的异常状态：

- **跟踪内容**：
  - 异常累积次数
  - 上次触发时间
  - 上次异常时间
- **自动清理**：定期清理24小时前的异常状态

## 配置参数

在 `/opt/netops/backend/config/site_config.py` 中配置：

```python
DEFAULT_SAMPLING_CONFIG = {
    # ... 其他配置 ...

    # 异常跟踪器配置
    "abnormal_tracker": {
        # 累积阈值：同一台设备的同一种异常需要连续多少次采样才触发研判
        "accumulation_threshold": 3,

        # 去重时间窗口（分钟）：同一台设备的同一种异常在此时间窗口内只触发一次
        "dedup_window_minutes": 30,

        # 冷却时间（分钟）：同一台设备的同一种异常触发后需要等待多久才能再次触发
        "cooldown_minutes": 60
    }
}
```

## 工作流程

1. **日志采样**：每5分钟采样一次日志
2. **异常检测**：检测每个设备是否异常
3. **异常跟踪**：
   - 检查是否在冷却期 → 如果是，跳过
   - 检查是否在去重时间窗口内 → 如果是，跳过
   - 增加异常计数
   - 检查是否达到累积阈值 → 如果是，触发研判
4. **研判处理**：执行完整的研判流程
5. **状态更新**：更新异常跟踪状态

## 预期效果

### 优化前

- 每5分钟采样一次
- 每次异常都触发研判
- 任务数量：~288个/天（24小时 × 12次/小时）

### 优化后（假设配置为3-30-60）

- 每5分钟采样一次
- 需要连续3次异常才触发研判（15分钟）
- 触发后冷却60分钟
- 任务数量：~24个/天（24小时 ÷ 1小时）

**减少约90%的任务数量**

## 使用建议

### 保守配置（适合生产环境）

```python
"abnormal_tracker": {
    "accumulation_threshold": 3,      # 连续3次异常才触发
    "dedup_window_minutes": 30,       # 30分钟内只触发一次
    "cooldown_minutes": 60            # 触发后冷却60分钟
}
```

### 激进配置（适合测试环境）

```python
"abnormal_tracker": {
    "accumulation_threshold": 2,      # 连续2次异常就触发
    "dedup_window_minutes": 15,       # 15分钟内只触发一次
    "cooldown_minutes": 30            # 触发后冷却30分钟
}
```

### 宽松配置（适合重要设备）

```python
"abnormal_tracker": {
    "accumulation_threshold": 5,      # 连续5次异常才触发
    "dedup_window_minutes": 60,       # 60分钟内只触发一次
    "cooldown_minutes": 120           # 触发后冷却120分钟
}
```

## 监控和调优

1. **监控指标**：
   - 每天触发的研判数量
   - 每个设备的异常累积次数
   - 冷却期跳过的异常数量

2. **调优建议**：
   - 如果任务数量仍然过多，增加累积阈值或冷却时间
   - 如果重要异常被忽略，减少累积阈值或冷却时间
   - 根据设备重要性使用不同的配置

## 注意事项

1. **内存使用**：异常跟踪器使用内存缓存，定期清理过期状态
2. **配置生效**：修改配置后需要重启服务
3. **数据持久化**：当前实现使用内存缓存，服务重启后状态丢失
4. **多实例部署**：多实例部署时需要使用分布式缓存（如Redis）

## 后续改进

1. **数据持久化**：将异常状态保存到数据库
2. **分布式缓存**：支持多实例部署
3. **设备级别配置**：为不同设备配置不同的阈值
4. **异常类型权重**：不同异常类型使用不同的阈值
5. **智能调优**：基于历史数据自动调整阈值
